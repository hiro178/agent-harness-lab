# tool-channel-resilience plugin

*[English](README.md) | [日本語](README_ja.md)*

劣化したツールトランスポートチャネル — AI エージェントとツール実行の間の層 — を生き延びるための規律ルール集。

## 問題

トランスポート自体が誤動作することがある: ツール結果が空や途中切れで返る、単純な `echo ok` プローブが何も返さない、ツール呼び出しのバッチ全体がまとめてキャンセルされる、あるいは応答がツール呼び出しの途中で止まる。エージェント（とユーザー）は本能的に「もっと強くリトライする」「ツール呼び出しには気をつけて、というプロンプト指示を加える」で対応しがちだが、どちらも効かない。**これはインフラの状態であり、振る舞いの問題ではない**からだ。

## テーゼ: 被害半径を制御する

指示だけでは不安定なトランスポートを信頼できるものにはできない。指示にできるのは、1回の障害がもたらすダメージの大きさを制御することだけだ:

- **小バッチ** — 1つの不正な呼び出しがそのバッチ全体をキャンセルする。バッチあたりの呼び出し数を減らせば被害半径も小さくなる。
- **バックグラウンド + ポーリング** — 重いコマンドを、脆弱な同期チャネルから完全に切り離す。
- **編集してから検証する** — `git checkout` 一発で復旧できるうちに、各編集後の静かなファイル破損を捕まえる。
- **永続状態をチェックポイントする** — チャネルが本当にダウンしているなら、盲目的にリトライするのではなく、存在するものをコミットしてユーザーに伝える。

このスキルには、検知チェックリスト（劣化したチャネルと通常のエラーの見分け方）、根拠付きの8つの規律ルール、macOS/BSD で動く background+poll の定番パターンが含まれる。

## インストール

```text
/plugin marketplace add hiro178/agent-harness-lab
/plugin install tool-channel-resilience@agent-harness-lab
```

## 発動タイミング

上記のパターンでツール呼び出しが失敗し始めたときにオンデマンドで — あるいはトランスポートの不調がコストの高い長時間自律セッションでは予防的に。
