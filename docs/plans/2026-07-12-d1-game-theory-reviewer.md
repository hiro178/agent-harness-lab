# D1: game-theory-reviewer — Builder vs Breaker 敵対的レビューフレームワーク

- 作成日: 2026-07-12 / 工数見積: 6 人日 / リスク: 高（勝負枠）
- 種別: エージェント仕様 + オーケストレーションスキル（新規開発）

## 目的

実装を担当する Builder エージェントと、そのコードを壊すことで「得点」する Breaker エージェントを競わせ、堅牢なコードとテストを自律生成する。単なる AI レビューを超えた敵対的インセンティブ設計で、実用性と学術的エンタメ性を両立する。9 ユニット中で最もバズ適性が高い。

## 設計概要

```text
/adversarial-build <spec>
  Round k:
    1. Builder: 仕様に対する実装（または前ラウンド指摘の修正）
    2. Breaker: 実装を攻撃 — 失敗する具体的テストケースを提出して得点
       （抽象的指摘は無得点。実行可能なテストのみが攻撃として成立）
    3. Judge（第三エージェント）: テストの正当性を判定
       — 仕様外の攻撃・自明でない仕様解釈は無効化
    4. 収束判定: Breaker が 2 ラウンド連続で有効打ゼロ → 終了
  ハードリミット: 最大 5 ラウンド / トークン予算上限（既定 200K、設定可能）
```

収束保証が最大の技術リスク。対策として全ゲートを機械的に敷く:

- **Verifier Theater 対策**: Breaker のテストは Builder が触れないディレクトリに隔離。Builder はテストを修正できない（コードのみ編集可）
- **オシレーション検知**: 同一箇所の修正が 2 ラウンド往復したら Judge が仕様曖昧性と判定し、人間へエスカレーション
- **無限ループ対策**: ラウンド上限 + トークン予算をハーネス側で強制（エージェントの自制に頼らない）

## 実装ステップ

1. Judge のルーブリック設計（有効打の定義）— ここが破綻すると全体が崩れるため最初に固める
2. Builder / Breaker / Judge のエージェント仕様を書く（Breaker には「refute せよ、不確実なら refuted に倒せ」の敵対バイアスを明示）
3. オーケストレーションスキル実装（ラウンド管理・状態ファイル・収束判定）
4. パイロット実験 3 本: 小さな仕様（例: rate limiter、日付パーサ、LRU キャッシュ）で実走し、ラウンド数・トークン消費・発見された欠陥数を記録
5. 実験記録を「What Didn't Work」テーブル込みで整理 — 収束しなかったケースも資産として公開する
6. README（英語）: 仕組み図 + パイロット実験の数値 + 制約の明記
7. plugin.json + marketplace.json 登録

## 撤退基準（事前定義）

パイロット 3 本中 2 本以上で「5 ラウンド以内に収束せず、かつ有効な欠陥発見もない」場合は、フレームワークとしての公開を中止し、**実験記録（research writeup）として公開に切り替える**。敵対的レビューの失敗分析はそれ自体が稀少なコンテンツであり、投下工数は無駄にならない。

## 注意点

- 工数 6 人日は 9 ユニット中最大 — A 群・B 群の公開が軌道に乗ってから着手する（先にこれを始めない）
- トークン消費が大きいことを README で正直に開示する（1 実行あたりの目安を実測値で示す）
- 類似研究（adversarial code review、GAN-inspired 分離）への言及と差分を README に書くと学術層からの流入が見込める
