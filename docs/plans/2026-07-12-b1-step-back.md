# B1: step-back — エラー反復ループ検知・脱出フック

- 作成日: 2026-07-12 / 工数見積: 2 人日 / リスク: 低
- 種別: フック + ルールのプラグイン（新規開発）

## 目的

AI 開発最大のストレスである「同じエラーを繰り返す無限ループ」を検知し、エージェントに「一歩引いた」前提再評価を強制する。試行の高速化ではなく、問い自体の再構成（「どう直すか」→「そもそもこのアプローチか」）を機械的に発火させる。

## 設計概要

```text
PostToolUse フック (Bash / Edit 対象)
  ├── ツール結果からエラーシグネチャを抽出（エラー種別 + 対象ファイルの正規化ハッシュ）
  ├── セッション状態ファイル（プロジェクト tmp/ 配下）に追記
  └── 同一シグネチャ 3 回目で additionalContext に step-back プロトコルを注入
        「試行を止めよ。試したアプローチと結果を列挙し、前提のどれが
         誤っている可能性が高いか評価してから次の手を選べ」
```

- しきい値（既定 3 回）と対象ツールは設定で変更可能にする
- 注入は 1 ループにつき 1 回（同じ注入の反復はノイズ）。脱出後（異なるシグネチャが N 回続いたら）カウンタをリセット

## 実装ステップ

1. エラーシグネチャ正規化の設計: 行番号・タイムスタンプ・一時パスを除去してから hash。ここが精度の核心
2. フックスクリプト実装（bash + jq、依存最小）。graceful degradation: jq 不在・JSON 不正・状態ファイル破損はすべて exit 0 + stderr 警告で素通しし、ユーザーの作業を絶対にブロックしない
3. step-back プロトコル文面の作成（列挙 → 前提評価 → 別角度の 3 段。長文にしない）
4. テスト: 同一エラーを 3 回発生させる再現スクリプトで発火を確認 + 異なるエラーが混在するケースで誤発火しないことを確認（敵対ケース）
5. B4 hook-testkit が完成していればそのフィクスチャで回帰テスト化する（B4 の最初の実利用例を兼ねる）
6. plugin.json（hooks 配線込み）+ marketplace.json 登録

## 注意点

- 「3 回失敗で問いを再構成」は A3 systematic-debugging の Phase 5 と同じ思想 — A3 は判断枠組み（advisory）、B1 はその機械的強制（enforcement）。この Formality Gradient の対比を README で明示すると、リポジトリ全体の思想の一貫性が伝わる
- false positive（テストの意図的失敗、TDD の Red フェーズ）を誤検知しない除外規則を設ける（例: pytest 実行直後の失敗は除外リスト対象）
